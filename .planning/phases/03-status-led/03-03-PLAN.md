  <what-built>
Status LED firmware driving GPIO15 active-low on XIAO ESP32-C6. Three blink patterns:
- Connecting (LED-01): 200ms on / 200ms off fast blink — while WiFi or MQTT connecting
- Connected (LED-02): steady on — when both WiFi and MQTT are connected
- Error (LED-03): 3x rapid pulse (100ms on/off each) then 700ms off, repeating — after 3+ max-backoff failures

State transitions:
- Boot → Connecting blink (immediately visible before WiFi connects)
- WiFi + MQTT both up → steady on
- WiFi or MQTT drops → returns to Connecting blink
- WiFi unreachable for 3+ min (3x at 60s backoff) → Error pattern
  </what-built>
  <how-to-verify>
**Verification A — LED-01 (Connecting blink):**
1. Power-cycle the device or watch boot from monitor
2. During the WiFi connection attempt, LED should blink at ~200ms on / 200ms off (fast blink, 5 Hz)
3. If WiFi is configured and connects quickly, you may need to temporarily disable your AP or change WIFI_SSID in config.rs to a wrong value to hold the device in Connecting state longer

**Verification B — LED-02 (Steady on):**
1. Once WiFi connects AND MQTT connects, LED goes steady on (no blinking)
2. Confirm via serial monitor: look for "MQTT connected" log message — LED should go steady shortly after
3. LED must be visually distinct from Connecting — no flicker, continuous illumination

**Verification C — LED-03 (Error pattern):**
Two options:
Option 1 (quickest): Temporarily set WIFI_PASS to wrong value in src/config.rs, cargo build, reflash. Watch for error pattern after ~3 min (3 failures at 60s backoff cap).
Option 2 (non-destructive): Disable your WiFi AP, wait for backoff to reach 60s (watch serial monitor for "Reconnecting in 60s"), observe 3+ consecutive failures, confirm error pattern.

Expected error pattern: LED pulses rapidly 3 times (100ms on, 100ms off each = 600ms total), then dark for 700ms, then repeats.

**Serial monitor hints:**
- "WiFi reconnect failed" + "Reconnecting in 60s..." = backoff at max
- After 3rd such message: LED should switch to Error pattern
- After WiFi recovers: LED returns to Connecting (not Error) while MQTT reconnects, then steady-on once MQTT fires Connected
  </how-to-verify>
  <resume-signal>Type "verified A B C" if all three patterns confirmed. Describe any deviation (e.g. "A ok, B ok, C not tested — no easy way to force error state" is acceptable if A+B are confirmed).</resume-signal>
