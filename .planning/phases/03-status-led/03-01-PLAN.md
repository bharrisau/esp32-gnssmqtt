**LedState enum:**
```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicU8, Ordering};
use esp_idf_hal::gpio::{Gpio15, Output, PinDriver};

#[repr(u8)]
#[derive(Clone, Copy, PartialEq)]
pub enum LedState {
    Connecting = 0,
    Connected  = 1,
    Error      = 2,
}

impl LedState {
    pub fn from_u8(v: u8) -> Self {
        match v {
            1 => LedState::Connected,
            2 => LedState::Error,
            _ => LedState::Connecting,
        }
    }
}
```

**led_task function** — takes exclusive ownership of the GPIO pin and the shared state Arc. Polls state every 50ms and drives blink timing via elapsed-time counter (NOT by sleeping for the full blink period, which would delay state changes).

Blink patterns per locked spec:
- Connecting (LED-01): 200ms on / 200ms off repeating
- Connected (LED-02): steady on (set_low once, no toggling)
- Error (LED-03): 3x rapid pulse (100ms on / 100ms off each) then 700ms off. Total cycle = 1300ms. Within the 1300ms cycle: position < 600ms AND (position % 200) < 100 = LED on.

Implementation approach: Use elapsed_ms counter incremented by 50 each poll tick. For Connecting and Error: use modular arithmetic on elapsed_ms. For Connected: track whether LED is already on to avoid redundant set_low calls. When state changes FROM Connected to anything else, reset elapsed_ms to 0 so blink starts cleanly.

Function signature:
```rust

```

Thread behavior: diverging (-> !), never returns. The thread is spawned in main.rs (Plan 03-03).

**State write logic inside the supervisor loop:**

1. After detecting WiFi disconnect (when !connected branch), write Connecting state before the backoff sleep:
   ```rust
   led_state.store(LedState::Connecting as u8, Ordering::Relaxed);
   ```

2. On successful WiFi reconnect (Ok(_) arm after wifi.connect() + wait_netif_up()):
   - Reset backoff_secs = 1
   - Reset max_backoff_failures counter to 0
   - Do NOT write Connected — that is the MQTT pump's responsibility (see anti-pattern in research)

3. Add local counter for error threshold tracking:
   ```rust
   let mut max_backoff_failures: u32 = 0;
   ```

4. On Err(_) reconnect arm, after doubling backoff:
   - If backoff_secs has reached 60 (at max): increment max_backoff_failures
   - If max_backoff_failures >= 3: write Error state
   ```rust
   if backoff_secs >= 60 {
       max_backoff_failures += 1;
       if max_backoff_failures >= 3 {
           led_state.store(LedState::Error as u8, Ordering::Relaxed);
       }
   }
   backoff_secs = (backoff_secs * 2).min(60);
   ```

5. On success path, reset max_backoff_failures = 0 (WiFi success resets the error counter, even though LED stays Connecting until MQTT fires Connected).

**CRITICAL — what wifi_supervisor must NOT do:**
- Do NOT write LedState::Connected — only the MQTT pump writes Connected on EventPayload::Connected
- The reason: WiFi reconnected != MQTT reconnected; writing Connected here would falsely show green before MQTT is up

The function still diverges (-> !). Preserve all existing log::warn!/log::error! calls.
