Initialization order in main() (MANDATORY — do not reorder):
1. esp_idf_svc::sys::link_patches()  ← first, always
2. EspLogger::initialize_default()   ← before any log:: calls
3. Peripherals::take()               ← take hardware ownership
4. EspSystemEventLoop::take()        ← required by WiFi
5. EspDefaultNvsPartition::take()    ← required by WiFi
6. wifi::wifi_connect(...)           ← WiFi BEFORE MQTT (IP required for TCP)
7. uart_bridge::spawn_bridge(...)    ← UART bridge (independent, can start after WiFi)
8. mqtt::mqtt_connect(...)           ← MQTT AFTER WiFi (TCP must be up)
9. Spawn pump thread (BEFORE any publish/subscribe)
10. Spawn heartbeat thread
11. Spawn wifi supervisor thread
12. Main thread: idle loop (or park)

main() body (follow this order exactly):

```rust
fn main() {
    // Step 1: linker patches — MUST be first
    esp_idf_svc::sys::link_patches();

    // Step 2: logging — MUST be before any log:: call
    EspLogger::initialize_default();

    let device_id = device_id::get();
    log::info!("=== esp32-gnssmqtt booting ===");
    log::info!("Device ID: {}", device_id);
    log::info!("Build: {} {}", env!("CARGO_PKG_NAME"), env!("CARGO_PKG_VERSION"));

    // Step 3: Take hardware peripherals
    let peripherals = Peripherals::take().expect("peripherals already taken");

    // Step 4: System event loop (required by WiFi)
    let sysloop = EspSystemEventLoop::take().expect("sysloop already taken");

    // Step 5: NVS partition (required by WiFi)
    let nvs = EspDefaultNvsPartition::take().expect("NVS already taken");

    // Step 6: WiFi — must be before MQTT (TCP requires IP)
    log::info!("Connecting to WiFi...");
    let wifi = wifi::wifi_connect(peripherals.modem, sysloop.clone(), nvs)
        .expect("WiFi connect failed");
    log::info!("WiFi connected");

    // Step 7: UART bridge (UM980 on UART1, GPIO20 RX / GPIO21 TX)
    uart_bridge::spawn_bridge(
        peripherals.uart1,
        peripherals.pins.gpio21,  // TX to UM980
        peripherals.pins.gpio20,  // RX from UM980
    ).expect("UART bridge init failed");
    log::info!("UART bridge started");

    // Step 8: MQTT — after WiFi (IP must be up)
    log::info!("Connecting to MQTT broker...");
    let (mqtt_client, mqtt_connection) = mqtt::mqtt_connect(&device_id)
        .expect("MQTT connect failed");
    log::info!("MQTT client created");

    // Step 9: Pump thread — MUST start before any publish/subscribe
    let pump_client = mqtt_client.clone();
    let pump_device_id = device_id.clone();
    std::thread::Builder::new()
        .stack_size(8192)
        .spawn(move || mqtt::pump_mqtt_events(mqtt_connection, pump_client, pump_device_id))
        .expect("pump thread spawn failed");

    // Step 10: Heartbeat thread
    let hb_client = mqtt_client.clone();
    let hb_device_id = device_id.clone();
    std::thread::Builder::new()
        .stack_size(8192)
        .spawn(move || mqtt::heartbeat_loop(hb_client, hb_device_id))
        .expect("heartbeat thread spawn failed");

    // Step 11: WiFi supervisor thread (reconnect on drop)
    std::thread::Builder::new()
        .stack_size(8192)
        .spawn(move || wifi::wifi_supervisor(wifi))
        .expect("wifi supervisor spawn failed");

    // Step 12: Main thread parks — all work is done in spawned threads
    log::info!("All subsystems started — device operational");
    loop {
        std::thread::sleep(std::time::Duration::from_secs(60));
    }
}
```

**CONN-05: Heartbeat with retain**
- Subscribe to `gnss/FFFEB5/heartbeat` on your broker (after ~35 seconds from boot)
- Expected: message payload "online", retain flag set
- Command: `mosquitto_sub -h <broker> -t "gnss/FFFEB5/heartbeat" -v`

**CONN-06: LWT — offline status on disconnect**
- Subscribe to `gnss/FFFEB5/status` before powering on the device
- Command: `mosquitto_sub -h <broker> -t "gnss/FFFEB5/status" -v`
- Power off the device (or disconnect its USB power)
- Expected: broker delivers "offline" to `gnss/FFFEB5/status` within ~70 seconds (keepalive interval)
- (May need to wait for keepalive timeout; check that the retain flag is set)

**CONN-03: WiFi reconnect after drop**
- While device is running and publishing heartbeats, temporarily disable your WiFi AP or change the SSID
- Serial monitor should show: WiFi disconnected → reconnecting with backoff → WiFi reconnected
- Re-enable WiFi AP (with correct SSID/pass) — device should reconnect and resume heartbeats without reboot

**CONN-04: MQTT reconnect after broker restart**
- While device is running, restart your MQTT broker (e.g., `sudo systemctl restart mosquitto`)
- Serial monitor should show: MQTT disconnected → MQTT connected — re-subscribing
- Device should resume heartbeat publishing without reboot

**CONN-07: UART bridge to UM980**
- With UM980 physically connected to XIAO ESP32-C6 GPIO20 (RX) / GPIO21 (TX):
  - Type a UM980 command in the USB serial console (e.g., `FRESET` or `VERSION`)
  - Observe UM980 response echoed back in the serial console
- If UM980 is not connected during this test: skip CONN-07 hardware verification and note it in the summary; the bridge code is correct per the build, hardware verification deferred to when UM980 is wired up
