**3. publish_thread function** — owns `EspMqttClient<'static>` exclusively; receives from `std::sync::mpsc::Receiver<MqttMessage>`. Use `recv_timeout(crate::config::RELAY_RECV_TIMEOUT)` loop pattern matching all existing relay threads.

Critical re-entrancy guard logic (per RESEARCH.md Pattern 3 and Pitfall 4):
- Set `crate::log_relay::LOG_REENTERING` to `true` ONLY when publishing a `MqttMessage::Log` variant.
- Clear it immediately after the `enqueue()` call for that variant.
- Do NOT set it for Nmea, Rtcm, Heartbeat, Status, or Bench variants — this would suppress the log relay nearly continuously at 40 msg/sec.

On `enqueue()` error: increment `MQTT_ENQUEUE_ERRORS` with `fetch_add(1, Ordering::Relaxed)`.

On `RecvTimeoutError::Timeout`: no-op (normal when no messages).

On `RecvTimeoutError::Disconnected`: log error and park (`loop { sleep(60s) }`).

**Thread stack note for callers (add as doc comment):** The publish thread stack should be 8192 bytes (same as nmea_relay and rtcm_relay). This is set in `main.rs` at spawn time, not here.

**HWM log at entry** (follow existing pattern in all relay threads):
```rust
let hwm_words = unsafe {
    esp_idf_svc::sys::uxTaskGetStackHighWaterMark(core::ptr::null_mut())
};
log::info!("[HWM] {}: {} words ({} bytes) stack remaining at entry",
    "MQTT publish", hwm_words, hwm_words * 4);
```

**Note on LOG_REENTERING visibility:** `log_relay::LOG_REENTERING` is currently not `pub`. This module accesses it as `crate::log_relay::LOG_REENTERING`. Plan 02 will make `LOG_REENTERING` pub in `log_relay.rs` — until then, add a `// TODO(plan02): make LOG_REENTERING pub in log_relay.rs` comment to mark the dependency, and write the access pattern assuming it will be pub. The `#[allow(dead_code)]` attribute may be needed temporarily until main.rs wiring (Plan 03) uses publish_thread.
