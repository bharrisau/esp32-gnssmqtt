**New signature:**
```rust
pub fn spawn_relay(
    mqtt_tx: std::sync::mpsc::SyncSender<crate::mqtt_publish::MqttMessage>,
    rtcm_topic: std::sync::Arc<str>,     // pre-built "gnss/{id}/rtcm"
    rtcm_rx: std::sync::mpsc::Receiver<crate::gnss::RtcmFrame>,
    free_pool_tx: std::sync::mpsc::SyncSender<Box<[u8; 1029]>>,
) -> anyhow::Result<()>
```

**Buffer management with bytes crate** (per RESEARCH.md Pattern 2):
- At thread entry, before the loop, allocate a working buffer: `let mut buf = bytes::BytesMut::with_capacity(1029);`
- Per frame: write frame bytes into buf, split and freeze into `Bytes`, send via channel.
- Call `buf.reserve(1029)` after `try_send` for best-effort reclaim.

```rust
Ok((message_type, frame_buf, frame_len)) => {
    // Write frame into BytesMut
    buf.extend_from_slice(&frame_buf[..frame_len]);
    let filled = buf.split().freeze();   // zero-copy handoff

    // Return pool buffer to GNSS RX thread (must happen regardless of publish success)
    if free_pool_tx.send(frame_buf).is_err() {
        log::error!("RTCM relay: free pool channel closed — buffer leaked");
    }

    // Publish via channel
    match mqtt_tx.try_send(crate::mqtt_publish::MqttMessage::Rtcm {
        topic: rtcm_topic.clone(),
        payload: filled,
    }) {
        Ok(()) => {}
        Err(std::sync::mpsc::TrySendError::Full(_)) => {
            log::warn!("RTCM relay: publish channel full — frame dropped");
            crate::gnss::RTCM_DROPS.fetch_add(1, std::sync::atomic::Ordering::Relaxed);
        }
        Err(std::sync::mpsc::TrySendError::Disconnected(_)) => {
            log::error!("RTCM relay: publish channel closed — thread exiting");
            break;
        }
    }

    // Best-effort reclaim: works if Bytes refcount has dropped to 1 (publish thread consumed it).
    // If not yet consumed, reserve() allocates fresh — acceptable, not a correctness issue.
    buf.reserve(1029);
}
```

**Topic consolidation:** RTCM was published to `gnss/{id}/rtcm/{message_type}`. Now publish all RTCM frames to `gnss/{id}/rtcm` (single topic). Message type is in the binary frame header — downstream consumers can parse it.

Remove the per-frame `format!("gnss/{}/rtcm/{}", device_id, message_type)` call. Use `rtcm_topic.clone()`.

On mutex-failure path (no longer exists — no mutex) and pool return on failure: pool buffer is returned BEFORE the try_send. If try_send fails, the frame is dropped but the pool buffer is already returned — no leak.

Remove `Arc<Mutex<EspMqttClient>>` imports. Add `bytes::{Bytes, BytesMut}`.

**2. src/mqtt.rs — update subscriber_loop to use direct EspMqttClient reference:**

Since `EspMqttClient` is now exclusively owned by the publish thread, `subscriber_loop` cannot call `client.subscribe()` via the publish channel (subscribe isn't an enqueue operation). Two options per RESEARCH.md Open Question 1:

Use the `Subscribe` variant approach to fully eliminate Arc/Mutex: Add `Subscribe { topic: String, qos: QoS, signal: SyncSender<()> }` to `MqttMessage` in `src/mqtt_publish.rs`, have the publish thread call `client.subscribe()` and send the signal, have `subscriber_loop` use the mqtt_tx channel.

Simpler option: keep subscriber_loop using a direct `EspMqttClient` reference by using a separate `Arc<Mutex<EspMqttClient>>` only for subscribe — but this reintroduces the mutex we're eliminating.

**Chosen approach per CONTEXT.md scope:** Add `Subscribe` variant to `MqttMessage`. Update `mqtt_publish::publish_thread` to handle it. `subscriber_loop` receives `SyncSender<MqttMessage>` and sends `MqttMessage::Subscribe` with a signal channel; blocks on the signal before returning.

Add to `MqttMessage` enum in `src/mqtt_publish.rs`:
```rust
Subscribe {
    topic: String,
    qos: embedded_svc::mqtt::client::QoS,
    signal: std::sync::mpsc::SyncSender<Result<(), String>>,
},
```

In `publish_thread` match arm:
```rust
MqttMessage::Subscribe { topic, qos, signal } => {
    let result = client.subscribe(&topic, qos).map_err(|e| format!("{:?}", e));
```

Update `subscriber_loop` signature to take `SyncSender<MqttMessage>` instead of `Arc<Mutex<EspMqttClient>>`. For each subscription, create a one-shot signal channel, send `MqttMessage::Subscribe`, and recv on the signal channel with a short timeout to confirm subscription.
