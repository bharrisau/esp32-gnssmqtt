<objective>
Fix BUG-1: DHCP DNS override not surviving wifi start in SoftAP mode.

The current code calls `esp_netif_dhcps_stop/set_dns_info/start` after `wait_netif_up()`, but ESP-IDF
reinitialises the DHCP server after wifi start, discarding all post-start DNS configuration. The correct
fix is to inject a pre-configured `EspNetif` via `EspWifi::wrap_all` before `wifi.start()`.

BUG-2 (Android captive portal 302) is already committed and requires no code change — it is unblocked
once BUG-1 is fixed and DNS resolves to the portal IP.

Purpose: Enable SoftAP clients to reach the captive portal without a manual IP navigation step.
Output: `provisioning.rs` without the unsafe DHCP block; `main.rs` using WifiDriver+wrap_all for SoftAP.
</objective>

Key research finding: `EspNetif::new_with_conf` with `RouterConfiguration { dns: Some(Ipv4Addr) }` calls
`set_dns()` AND `esp_netif_dhcps_option(OFFER_DNS)` during construction — the only lifecycle point that
survives wifi start. Source verified from esp-idf-svc-0.51.0/src/netif.rs lines 370-457.

`run_softap_portal` currently accepts `wifi: &mut BlockingWifi<EspWifi<'static>>` and expects WiFi to
be unconfigured (it calls `wifi.set_configuration(...)` and `wifi.start()` itself). The change moves
WiFi construction from main.rs into provisioning.rs OR keeps construction in main.rs but builds with
wrap_all. The cleaner approach (per research): keep BlockingWifi construction in main.rs, but change the
construction call from `EspWifi::new(...)` to `WifiDriver::new + EspWifi::wrap_all(driver, sta, ap_netif)`.
</context>

Replace the current `EspWifi::new(...)` call (lines ~126-130) with the three-step WifiDriver+wrap_all pattern:

```rust
// Add imports at top of the existing esp_idf_svc::wifi use block (or inline):
use esp_idf_svc::wifi::WifiDriver;
use esp_idf_svc::netif::{EspNetif, NetifConfiguration, NetifStack};
use esp_idf_svc::ipv4::{
    Configuration as IpConfiguration, RouterConfiguration, Subnet, Mask,
};
use std::net::Ipv4Addr;

// Construction (replaces lines 126-130):
let ap_netif = EspNetif::new_with_conf(&NetifConfiguration {
    ip_configuration: Some(IpConfiguration::Router(RouterConfiguration {
        subnet: Subnet {
            gateway: Ipv4Addr::new(192, 168, 71, 1),
            mask: Mask(24),
        },
        dhcp_enabled: true,
        dns: Some(Ipv4Addr::new(192, 168, 71, 1)),  // portal IP as DNS — NOT default 8.8.8.8
        secondary_dns: None,
        ..Default::default()
    })),
    ..NetifConfiguration::wifi_default_router()  // preserves WIFI_AP_DEF key/desc/stack/events
})?;
let softap_driver = WifiDriver::new(peripherals.modem, sysloop.clone(), Some(nvs.clone()))
    .expect("WifiDriver new failed in SoftAP path");
let softap_esp_wifi = EspWifi::wrap_all(
    softap_driver,
    EspNetif::new(NetifStack::Sta)?,
    ap_netif,
).expect("EspWifi wrap_all failed in SoftAP path");
let mut softap_wifi = esp_idf_svc::wifi::BlockingWifi::wrap(softap_esp_wifi, sysloop.clone())
    .expect("BlockingWifi wrap failed");
```

The `?` operator cannot be used inside an `if` branch that returns a value diverging to `unreachable!()`.
Use `.expect(...)` for the wrap_all call (consistent with existing error handling in this branch).
However `EspNetif::new_with_conf` returns `Result` — either use `?` (if the outer fn returns Result,
which main() does), or `.expect(...)`. Use `.expect(...)` consistently since the SoftAP path uses
`expect` throughout.

After this change, the `use` imports in main.rs may need updating. The existing `use esp_idf_svc::wifi::{BlockingWifi, EspWifi}` must gain `WifiDriver`. The ipv4/netif types may need separate `use` lines — add them at the top of main.rs in the existing import section.
