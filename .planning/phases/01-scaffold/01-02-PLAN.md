<objective>
Flash the compiled firmware to the physical ESP32-C6 hardware, observe the device ID printed to serial, and verify that the ID is stable across power cycles. This is the only checkpoint in Phase 1 — it proves that the scaffold works on real hardware, not just in the compiler.

Purpose: A firmware that compiles but does not boot correctly on hardware is useless. This plan closes the gap between "builds" and "works on device."

Output: Confirmed evidence that the device ID reads correctly from eFuse and is stable across reboots. The operator now knows this device's ID string, which will appear in every MQTT topic path.
</objective>

<task type="auto">
  <name>Task 1: Flash firmware to hardware</name>
  <files></files>
  <action>
Connect the XIAO ESP32-C6 to the development machine via USB. Then flash and open the serial monitor:

```bash
cargo run 2>&1
```

This runs `espflash flash --monitor` (configured in `.cargo/config.toml` as the runner for the `riscv32imac-esp-espidf` target).

Expected serial output after boot:
```
I (xxx) esp32_gnssmqtt: === esp32-gnssmqtt booting ===
I (xxx) esp32_gnssmqtt: Device ID: XXXXXX
I (xxx) esp32_gnssmqtt: Build: esp32-gnssmqtt 0.1.0
```

Where `XXXXXX` is a 6-character uppercase hex string (e.g. `A1B2C3`).

If `cargo run` fails with "No serial ports found" or similar:
- Verify the USB cable is connected and the device is powered
- Run `espflash board-info` to detect the device
- On Windows: check Device Manager for the COM port; pass `--port COMx` to espflash if needed by updating the runner in `.cargo/config.toml` to `runner = "espflash flash --monitor --port COMx"`

If the device boots but the log shows `esp_efuse_mac_get_default failed`:
- This indicates eFuse CRC corruption — a hardware defect. Try a different device.
- Do not proceed with a device that fails this check.

Record the device ID string printed to serial. This string is the permanent identifier for this hardware unit.
  </action>
  <verify>
    <automated>cargo run 2>&1 | grep -E "Device ID: [0-9A-F]{6}" || echo "FAIL: Device ID not found in output"</automated>
  </verify>
  <done>The serial output contains a line matching `Device ID: [0-9A-F]{6}`. The 6-character hex ID is visible in the monitor output. espflash exited without errors.</done>
</task>

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Human verify stable device ID across power cycles</name>
  <files></files>
  <action>Human verification step — power-cycle the device and confirm the device ID is identical across reboots. See how-to-verify below.</action>
  <verify>
    <automated>MISSING — human power-cycle verification required; no automated equivalent exists</automated>
  </verify>
  <done>Device ID is identical across at least 2 power cycles with no panics or watchdog errors in the boot log.</done>
  <what-built>
    The firmware has been flashed and the device is printing its hardware-derived device ID to serial. Claude observed the device ID in the serial output from Task 1.
  </what-built>
  <how-to-verify>
    1. Note the Device ID string displayed in the serial monitor from Task 1 (e.g., "A1B2C3").
    2. Power-cycle the ESP32-C6: disconnect USB, wait 5 seconds, reconnect.
    3. Open the serial monitor again: run `espflash monitor` (or `cargo run` will reflash and monitor).
    4. Observe the boot log. Confirm:
       - The line `Device ID: XXXXXX` appears where XXXXXX is the same 6-character string as before power cycle.
       - The heartbeat line `Heartbeat — Device ID: XXXXXX` appears every 5 seconds with the same ID.
    5. Power-cycle a second time and repeat the check.

    Pass criteria:
    - Device ID is identical across all power cycles observed
    - Device ID is exactly 6 uppercase hex characters
    - No panic or error messages in the boot log
    - FreeRTOS task watchdog does not trigger (no "Task watchdog got triggered" in output)
  </how-to-verify>
  <resume-signal>Type "approved" if device ID is stable and matches across power cycles. Describe any issues if the ID changes or errors appear.</resume-signal>
</task>
