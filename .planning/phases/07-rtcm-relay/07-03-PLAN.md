  <action>
Make three targeted changes to `src/main.rs`:

**Change 1: Add mod declaration.**
In the mod block (after `mod nmea_relay;`), add:
```rust
mod rtcm_relay;
```

**Change 2: Destructure three values from spawn_gnss.**
Update the spawn_gnss call site from:
```rust
let (gnss_cmd_tx, nmea_rx) = gnss::spawn_gnss(
    peripherals.uart0,
    peripherals.pins.gpio16,
    peripherals.pins.gpio17,
)
.expect("GNSS init failed");
```
to:
```rust
// gnss::spawn_gnss now returns three values: cmd_tx, nmea_rx, rtcm_rx
let (gnss_cmd_tx, nmea_rx, rtcm_rx) = gnss::spawn_gnss(
    peripherals.uart0,
    peripherals.pins.gpio16,  // TX line to UM980
    peripherals.pins.gpio17,  // RX line from UM980
)
.expect("GNSS init failed");
```

**Change 3: Spawn the RTCM relay thread after the NMEA relay.**
After the existing nmea_relay::spawn_relay call and its log::info, add:
```rust
// Step 16: RTCM relay — consumes rtcm_rx, publishes each verified RTCM3 frame to MQTT.
// rtcm_rx is moved into spawn_relay — do NOT retain a reference here.
// IMPORTANT: rtcm_rx is the THIRD element from spawn_gnss; nmea_rx is the SECOND.
// Passing them in the wrong order would compile (same Vec<u8> inner type) but publish
// NMEA as binary and RTCM as text — name variables clearly to avoid this pitfall.
rtcm_relay::spawn_relay(mqtt_client.clone(), device_id.clone(), rtcm_rx)
    .expect("RTCM relay thread spawn failed");
log::info!("RTCM relay started");
```

**Change 4: Update the initialization order comment** at the top of main() to add step 16:
```
//! 16. RTCM relay: spawn_relay(mqtt_client clone, device_id clone, rtcm_rx)
```

Do NOT change anything else — the config_relay call (step 15), the idle loop, or any other subsystem.
