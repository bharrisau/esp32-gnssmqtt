<objective>
Fix the latent topic-routing bug in `pump_mqtt_events` and bump the MQTT output buffer to 2048 bytes.

Purpose: The current `EventPayload::Received` arm ignores the `topic` field and forwards ALL received payloads to `config_tx`, meaning a future `/ota/trigger` message would be sent verbatim to the UM980 as a command. This fix makes routing correct. The buffer bump (1024 → 2048) ensures RTCM MSM7 frames up to 1029 bytes publish without truncation.

Output: Modified `src/mqtt.rs` with two changes — topic discrimination in `pump_mqtt_events` and `out_buffer_size: 2048` in `mqtt_connect`.
</objective>

  <name>Task 1: Fix topic discrimination in pump_mqtt_events (RTCM-05)</name>
  <files>src/mqtt.rs</files>
  <action>
In `pump_mqtt_events`, replace the `EventPayload::Received { data, .. }` arm with topic-aware routing:

```rust
EventPayload::Received { topic, data, .. } => {
    // topic: Option<&str> — None on chunked subsequent frames; Some(...) for complete messages.
    // Config and OTA payloads arrive as Details::Complete (single chunk), so topic is always Some here.
    let t = topic.unwrap_or("");
    if t.ends_with("/config") {
        // Forward config payloads to config_relay → UM980 UART
        match config_tx.send(data.to_vec()) {
            Ok(_) => {}
            Err(e) => log::warn!("Config relay channel closed: {:?}", e),
        }
    }
    // /ota/trigger and all other topics: silently ignored here.
    // Phase 8 will add ota_tx channel and route /ota/trigger payloads.
}
```

Update the doc comment above `pump_mqtt_events` to reflect the new routing behaviour — replace "On `Received`, copies data to a Vec and sends it through config_tx" with a description of topic-based dispatch.

Do NOT add an `else` log for unrecognised topics — this would log every /ota/trigger retain playback during Phase 8 development. Silent drop is intentional.
