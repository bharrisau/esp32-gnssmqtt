  <name>Task 1: Build canary firmware image for OTA validation</name>
  <files>src/main.rs</files>
  <action>
Make a minimal, distinguishable change to produce a "canary" firmware image that proves the OTA-delivered image differs from the currently running image. The simplest approach: add a startup log line in main.rs that includes a version marker.

In `src/main.rs`, find the first `log::info!` call in `fn main()` and add immediately after it:

```rust
log::info!("esp32-gnssmqtt v2.0-ota-canary — OTA validation build");
```

This single log line makes the new image verifiable: after OTA reboot, this line appears in the log stream (MQTT or serial) confirming the new image is running.

Build the release binary:
```
cargo build --release
```

Binary path: `target/xtensa-esp32-espidf/release/esp32-gnssmqtt`

Convert to flashable binary using esptool or the ESP-IDF elf2image tool (the build system may do this automatically — check if a .bin file is generated in the release dir):
```
ls target/xtensa-esp32-espidf/release/esp32-gnssmqtt.bin 2>/dev/null || \
  ls target/xtensa-esp32-espidf/release/*.bin 2>/dev/null
```

If no .bin is present, use esptool.py to generate it:
```
esptool.py --chip esp32 elf2image --output esp32-gnssmqtt.bin \
  target/xtensa-esp32-espidf/release/esp32-gnssmqtt
```

Compute SHA-256 of the binary:
```
sha256sum esp32-gnssmqtt.bin
```

Record the hex digest — needed for the OTA trigger payload.

Run `cargo clippy -- -D warnings` to confirm the canary log line does not introduce any warnings (it should not).

IMPORTANT: Do NOT commit the canary log line to the main branch. This is a temporary build artifact for validation. After hardware validation succeeds, revert the canary log line before creating the phase completion commit. The canary line may optionally be left as a permanent version marker at the developer's discretion — check with user in Task 2 checkpoint.

  <how-to-verify>
**Setup:**
- Device FFFEB5 connected to WiFi and MQTT broker
- Development machine on the same WiFi network
- Binary from Task 1 available; SHA-256 recorded

**Part A — OTA Firmware Update (MAINT-03):**

1. Serve the binary over HTTP from the development machine:
   ```
   cd /path/to/binary
   python3 -m http.server 8080
   ```

2. Verify the HTTP server is reachable from the device's network:
   ```
   curl -o /dev/null -s -w "%{http_code}" http://192.168.X.X:8080/esp32-gnssmqtt.bin
   ```
   Expected: `200`. If firewall blocks: `sudo ufw allow 8080`

3. Subscribe to device status topics to observe OTA progression:
   ```
   Topic: gnss/FFFEB5/ota/status
   Topic: gnss/FFFEB5/log
   Topic: gnss/FFFEB5/heartbeat
   ```

4. Publish the OTA trigger:
   ```
   Topic:   gnss/FFFEB5/ota/trigger
   Payload: {"url":"http://192.168.X.X:8080/esp32-gnssmqtt.bin","sha256":"<hex-from-task1>"}
   Retain:  true
   QoS:     1
   ```

5. Observe the following sequence:
   - `/ota/status` shows `downloading` (then progress if implemented)
   - `/ota/status` shows `complete` or device reboots
   - After reboot: `/log` shows canary line `esp32-gnssmqtt v2.0-ota-canary`
   - `/log` shows `mark_running_slot_valid` — confirms slot marked VALID (no rollback)
   - Device reconnects to MQTT and publishes a heartbeat — confirms it stays up

6. Clear the retained OTA trigger (prevents re-flash on reconnect):
   ```
   Topic:   gnss/FFFEB5/ota/trigger
   Payload: (empty)
   Retain:  true
   ```

**Part B — Captive Portal SoftAP Detection (deferred from Phase 17):**

1. On device FFFEB5, trigger SoftAP mode via one of:
   - Hold GPIO9 low for 3 seconds, OR
   - Publish `"softap"` to `gnss/FFFEB5/ota/trigger`

2. On a mobile device (iOS or Android), connect to the `GNSS-Setup` WiFi network (open, no password).

3. Verify the mobile device shows a captive portal notification or automatically opens the provisioning web UI.

4. Open the provisioning web UI in the mobile browser (navigate to the device IP if not auto-opened).

5. Confirm the form is accessible and renders correctly on mobile.

6. Allow the 300-second no-client timeout to restore normal WiFi/MQTT operation, OR manually reboot the device.
