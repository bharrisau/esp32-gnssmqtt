<task type="auto">
  <name>Task 1: Redesign partitions.csv for dual-slot OTA layout</name>
  <files>partitions.csv</files>
  <action>
Replace the current factory-only partition table with a dual-slot OTA layout that fits within 4MB flash.

Current file has: nvs (0x9000, 0x10000), phy_init (0x19000, 0x1000), factory (0x20000, 0x3E0000).

Replace with the following content exactly:

```
# Name,     Type, SubType,  Offset,   Size,    Flags
nvs,        data, nvs,      0x9000,   0x6000,
otadata,    data, ota,      0xF000,   0x2000,
ota_0,      app,  ota_0,    0x20000,  0x1E0000,
ota_1,      app,  ota_1,    0x200000, 0x1E0000,
```

Layout math (all within 4MB = 0x400000):
- nvs: 0x9000 → 0xF000 (shrunk from 0x10000 to 0x6000 to make room for otadata)
- otadata: 0xF000 → 0x11000 (0x2000 = 8KB; two 4KB OTA select sectors)
- ota_0: 0x20000 → 0x200000 (0x1E0000 = 1,966,080 bytes ≈ 1.875 MB)
- ota_1: 0x200000 → 0x3E0000 (0x1E0000 = same size)
- Slack: 0x3E0000 → 0x400000 (128 KB unused — within 4MB boundary)

Remove the phy_init row — ESP-IDF v5 embeds phy calibration data in NVS; a separate phy partition is not required on ESP32-C6.

The sdkconfig.defaults CONFIG_PARTITION_TABLE_CUSTOM_FILENAME path must remain unchanged — it references partitions.csv via the existing relative path.
  </action>

  </verify>
  <done>partitions.csv contains otadata, ota_0, and ota_1 rows with correct offsets; factory row removed; file fits 4MB layout math</done>

<task type="auto">
  <name>Task 2: Add rollback config + watchdog extension to sdkconfig.defaults; add sha2 to Cargo.toml</name>
  <files>sdkconfig.defaults, Cargo.toml</files>
  <action>
**sdkconfig.defaults:** Append the following two lines at the end of the file (after the existing UART RX ring buffer note):

```
# OTA: enable bootloader rollback — new firmware must call mark_running_slot_valid()
# or the bootloader rolls back to the previous slot on the next reboot.
CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE=y

# OTA: extend task watchdog to 30s to survive partition erase (takes 4-8s on ESP32-C6).
# The default 5s fires during EspOta::initiate_update() which erases the full ~1.875MB slot.
CONFIG_ESP_TASK_WDT_TIMEOUT_S=30
```

**Cargo.toml:** Add sha2 to [dependencies]. The crate is already present in the local registry at version 0.10.9 and is pure Rust / no_std+alloc compatible. Add this line after the existing `log = "0.4"` entry:

```toml
sha2 = { version = "0.10", default-features = false, features = ["oid"] }
```

Note: `default-features = false` omits the `std` feature which is not available in the ESP-IDF embedded target. The `oid` feature is benign (OID constants) and the only non-default feature that is no_std-compatible. If the linker rejects `oid`, drop it entirely — `sha2 = { version = "0.10", default-features = false }` also works. The core `Digest` + `Sha256` API is always available regardless of features.

After both edits, run `cargo build --release` to confirm the sha2 dependency resolves and the new sdkconfig keys are accepted without errors.
