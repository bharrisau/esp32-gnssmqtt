# Plan 09-02: Loop Hardening + recv_timeout Conversion

**Phase**: 9 — Channel + Loop Hardening
**Requirements**: HARD-05, HARD-06
**Depends on**: Plan 09-01 complete (channels are now sync_channel; SyncSender types in place)

## Goal

Replace all unbounded blocking `recv()` calls (via `for x in &receiver` iterator) with
`recv_timeout()` loops with documented maximum wait durations. Add an explicit max-retry constant
to the WiFi supervisor reconnect loop.

## Problem Statement

Every `for x in &receiver` pattern calls `Receiver::recv()` internally, which blocks the thread
indefinitely if no item arrives. If the producer thread stalls or dies, the consumer thread hangs
silently with no timeout.

Current unbounded blocking receives:
| Location | Receiver | Thread |
|----------|----------|--------|
| `gnss.rs` TX thread | `cmd_rx` | gnss-tx |
| `nmea_relay.rs:38` | `nmea_rx` | nmea-relay |
| `rtcm_relay.rs:37` | `rtcm_rx` | rtcm-relay |
| `config_relay.rs:36` | `config_rx` | config-relay |
| `mqtt.rs:148` | `subscribe_rx` | subscriber |
| `ota.rs:68` | `ota_rx` | ota |

The WiFi supervisor (`wifi.rs`) has a backoff counter (`max_backoff_failures`) but no named constant
for the limit, making the threshold implicit.

## Pattern: for-loop → recv_timeout loop

Replace:
```rust
for item in &receiver {
    // process item
}
log::error!("channel closed");
loop { std::thread::sleep(Duration::from_secs(60)); }
```

With:
```rust
loop {
    match receiver.recv_timeout(RECV_TIMEOUT) {
        Ok(item) => {
            // process item
        }
        Err(RecvTimeoutError::Timeout) => {
            // No item within timeout — normal if producer is quiet (GNSS idle, config rare).
            // Continue polling; this is the watchdog-feed opportunity in Phase 11.
        }
        Err(RecvTimeoutError::Disconnected) => {
            log::error!("...: channel closed — thread exiting");
            break;
        }
    }
}
// Dead-end park (producer exited; thread has nothing to do).
loop { std::thread::sleep(Duration::from_secs(60)); }
```

## Changes Required

### Shared timeout constants in `src/config.rs`

Add named timeout constants so all threads use consistent, documented values:

```rust
/// Blocking channel receive timeout for hot-path relay threads (NMEA, RTCM, GNSS TX).
/// At 10 Hz NMEA rate, timeouts occur only if GNSS pipeline stalls — used as a liveness check.
pub const RELAY_RECV_TIMEOUT: std::time::Duration = std::time::Duration::from_secs(5);

/// Blocking channel receive timeout for low-rate threads (config relay, OTA, subscriber).
/// These threads receive events rarely; timeout simply prevents indefinite hang if producer dies.
pub const SLOW_RECV_TIMEOUT: std::time::Duration = std::time::Duration::from_secs(30);

/// Maximum consecutive WiFi reconnect failures before logging an error.
/// After this many consecutive failures, LedState::Error is set.
/// Phase 12 (RESIL-01) will add esp_restart() at this threshold.
pub const MAX_WIFI_RECONNECT_ATTEMPTS: u32 = 20;
```

### 1. `src/gnss.rs` — TX thread

Change the TX thread loop from `for line in &cmd_rx` (after Plan 09-01 may use `.iter()`):

```rust
use std::sync::mpsc::RecvTimeoutError;
use crate::config::RELAY_RECV_TIMEOUT;

// TX thread body:
loop {
    match cmd_rx.recv_timeout(RELAY_RECV_TIMEOUT) {
        Ok(line) => {
            if let Err(e) = uart_tx.write(line.as_bytes()) {
                let n = UART_TX_ERRORS.fetch_add(1, Ordering::Relaxed) + 1;
                log::warn!("GNSS TX: write error #{}: {:?}", n, e);
            }
            if let Err(e) = uart_tx.write(b"\r\n") {
                let n = UART_TX_ERRORS.fetch_add(1, Ordering::Relaxed) + 1;
                log::warn!("GNSS TX: CRLF write error #{}: {:?}", n, e);
            }
        }
        Err(RecvTimeoutError::Timeout) => {
            // No command within 5s — normal during idle operation. Continue.
        }
        Err(RecvTimeoutError::Disconnected) => {
            log::error!("GNSS TX: cmd channel closed — TX thread exiting");
            break;
        }
    }
}
```

### 7. `src/wifi.rs` — reconnect loop hardening (HARD-05)

Add a named constant and proper per-failure counter (not just at-max-backoff counter):

```rust
use crate::config::MAX_WIFI_RECONNECT_ATTEMPTS;

pub fn wifi_supervisor(mut wifi: BlockingWifi<EspWifi<'static>>, led_state: Arc<AtomicU8>) -> ! {
    let mut backoff_secs: u64 = 1;
    let mut consecutive_failures: u32 = 0;  // replaces max_backoff_failures

    loop {
        std::thread::sleep(std::time::Duration::from_secs(5));

        let connected = wifi.is_connected().unwrap_or(false);

        if !connected {
            log::warn!("WiFi disconnected. Reconnecting in {}s (attempt {}/{})...",
                backoff_secs, consecutive_failures + 1, MAX_WIFI_RECONNECT_ATTEMPTS);

            led_state.store(LedState::Connecting as u8, Ordering::Relaxed);
            std::thread::sleep(std::time::Duration::from_secs(backoff_secs));

            let reconnect_ok = match wifi.connect() {
                Err(e) => {
                    log::error!("WiFi reconnect failed: {:?}", e);
                    false
                }
                Ok(_) => match wifi.wait_netif_up() {
                    Ok(_) => true,
                    Err(e) => {
                        log::error!("WiFi netif up failed after reconnect: {:?}", e);
                        false
                    }
                }
            };

            if reconnect_ok {
                log::info!("WiFi reconnected");
                backoff_secs = 1;
                consecutive_failures = 0;
            } else {
                consecutive_failures += 1;
                if consecutive_failures >= MAX_WIFI_RECONNECT_ATTEMPTS {
                    // Phase 12 (RESIL-01) will call esp_restart() here.
                    // For now, log error and set LED error state.
                    log::error!(
                        "WiFi: {} consecutive reconnect failures — giving up (will restart in Phase 12)",
                        consecutive_failures
                    );
                    led_state.store(LedState::Error as u8, Ordering::Relaxed);
                }
                backoff_secs = (backoff_secs * 2).min(60);
            }
        }
    }
}
```

This refactoring:
- Replaces implicit `max_backoff_failures` (only counted at max backoff) with
  `consecutive_failures` (counts every failure, reset on success)
- Uses `MAX_WIFI_RECONNECT_ATTEMPTS` as a named constant (20)
- Logs the attempt count so operators can see the progression in logs
- Leaves a Phase 12 comment at the threshold point for `esp_restart()` insertion

## Notes

- `lock()` calls in nmea_relay and rtcm_relay hot paths are effectively bounded: `enqueue()` is
  non-blocking (writes to MQTT outbox buffer and returns), so mutex hold time is < 1ms. Changing
  these to `try_lock()` would add complexity for negligible safety gain. Documented here as
  reviewed and accepted.
- The `-> !` return type is preserved on all affected functions — the dead-end park loop at the end
  satisfies it.
- `recv_timeout` Timeout arm is a no-op (just `continue`) in all threads. Phase 11 (Thread
  Watchdog) will feed heartbeat counters in the Timeout arm as a "thread is alive" signal.
