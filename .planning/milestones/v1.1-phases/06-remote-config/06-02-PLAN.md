(Or equivalent flash + monitor command for your setup.)

**Step 2 — Observe boot logs.** Expect to see:
- `Config relay started` in the boot sequence (after `NMEA relay started`)

**Step 3 — Verify CONF-01 (subscribe + forward).** Publish a JSON config payload:
```

  -t 'gnss/FFFEB5/config' -r \
  -m '{"delay_ms": 200, "commands": ["MODE ROVER", "CONFIGSAVE"]}'
```
Expect in espflash monitor:
- `Config relay: new config payload, hash 0x........`
- `Config relay: sending command: "MODE ROVER"`
- (approximately 200ms pause)
- `Config relay: sending command: "CONFIGSAVE"`

**Step 4 — Verify CONF-02 (hash dedup).** Power cycle the device OR force a WiFi/MQTT reconnect. After reconnect:
- Expect: `Config relay: payload unchanged (hash 0x........), skipping`
- Config must NOT be re-applied (no "sending command" lines appear)

**Step 5 — Verify CONF-03 (per-command delay override).** The delay_ms:200 in Step 3 should produce ~200ms between the two command log lines. Verify by eye in espflash monitor.

**Step 6 — Verify plain text fallback.** Publish without JSON:
```
mosquitto_pub -h <host> -u <user> -P <pass> \
  -t 'gnss/FFFEB5/config' -r \
  -m "MODE ROVER
CONFIGSAVE"
```
Expect: both commands forwarded with ~100ms default delay between them.

**Step 7 — Verify empty payload guard.** Clear the retained message:
```
mosquitto_pub -h <host> -u <user> -P <pass> \
  -t 'gnss/FFFEB5/config' -r -n
```
Expect: `Config relay: empty payload — retained message cleared, skipping` — no commands forwarded.
